# 下一阶段任务清单

## 1. 当前可用状态

当前项目已经完成 MVP 验收闭环：

- UE5.6 工程可以加载 `LocalTTS` 插件。
- UE 可以启动本地 FastAPI TTS 服务。
- 本地服务可以加载 `OmniVoice`。
- UE 可以调用 `/health`。
- UE 可以调用 `/tts`。
- 服务可以生成 wav 文件。
- UE 可以读取 `LastGeneratedWavPath`。
- UE 内实际播放链路已通过测试。
- `design` 模式配置链路已通过测试，可以按合法 `Instruct` 标签生成和播放。
- 具体音色需要按项目、角色和场景继续匹配。

最近一次有效结果示例：

```text
D:\project\UETTsProject\Services\tts_service\cache\req_000003.wav
```

## 2. 下一步优先级

下一阶段优先做产品化，不急着做 MetaHuman、口型、流式播放。

推荐顺序：

1. 蓝图正式接口整理。

5. 数字人联动预留。
6. 长文本分段与队列播放。
7. 编译、安装、交付流程。

## 5. P3：音频播放稳定性

目标：让 UE 内播放成为稳定能力。

任务：

- [x] 确认 wav 加载方式是否适合运行时文件。
- [x] 播放开始、播放完成、播放失败都有回调。
- [x] `Stop` 可以中断当前播放。
- [x] 支持 2D 播放。
- [x] 支持绑定到 Actor 的 3D 播放。
- [x] 支持只生成 wav、不自动播放，给数字人口型和表情系统使用。
- [x] 清理无用 AudioComponent。
- [x] 处理 wav 文件不存在、格式不支持、音频设备不可用。
- [x] 播放开始、停止、完成日志可见。
- [x] UE 日志输出 request_id，便于和服务日志对齐。
- [x] 将“生成 WAV”和“播放 WAV”拆成独立蓝图入口。
- [x] 增加播放最新 WAV、播放指定 WAV、播放历史响应、播放语音事件的异步播放节点。
- [x] 增加最近生成响应历史，支持 UI 重播旧声音。

当前实现结论：

- 运行时 wav 加载方式固定为：从磁盘读取服务端输出的 wav 文件，校验后加载为 `USoundWaveProcedural` 再播放。
- 当前正式支持的运行时播放输入是 `16-bit PCM`、`mono/stereo`、存在有效 sample data 的 wav 文件。
- 如果 wav 文件不存在、格式不是受支持的 PCM wav、位深不对、声道数异常，或当前世界没有可用音频设备，都会在播放前给出明确错误信息。

验收标准：

- 连续生成并播放 3 次不会崩溃。
- `Stop` 能立刻停止当前播放。
- 蓝图能收到播放完成事件。
- 蓝图可以通过 `Generate Local TTS Async` 拿到 wav 路径，而不触发 UE 内置播放。
- 测试 Actor 可以通过 `Gen` 按钮验证只生成不播放链路。
- 手动制造坏 wav / 缺失 wav / 无音频设备场景时，`OnError` 或调试字段能看到明确原因。

## 7. P5：数字人联动预留

目标：插件不只服务于“播放音频”，还要能服务于数字人口型、表情和表演驱动。

当前结论：

- 短期不直接实现 MetaHuman、口型识别或表情系统。
- 现在必须先把“语音生成”和“音频播放”解耦，避免后面数字人只能从播放器里反向取数据。
- `Generate Local TTS Async` 是数字人链路的推荐入口：它只生成 wav 和响应数据，不自动播放。
- `Speak Local TTS Async` 继续面向普通 2D 播放。
- `Speak Local TTS At Actor Async` 面向场景内 Actor 的 3D 播放。

后续任务：

- [x] 增加只生成 wav、不自动播放的蓝图节点。
- [x] 增加统一语音事件结构体 `FLocalTTSSpeechEvent`。
- [x] 在语音事件中保留 `RequestId`、`WavPath`、`Text`、`DurationSeconds` 等字段。
- [x] 预留 Viseme 口型轨道字段。
- [x] 预留 Emotion 表情轨道字段。
- [x] 异步语音节点增加 `OnSpeechEventReady` 回调。
- [ ] 设计数字人驱动接口，例如 `ULocalTTSDigitalHumanDriver`。
- [ ] 调研 UE/MetaHuman 可接入口型方案，再决定是否接第三方 lip sync 插件或自研分析。

建议链路：

```text
OmniVoice 生成 wav
-> Generate Local TTS Async 返回响应
-> 数字人系统读取 wav/text/request_id
-> 口型分析或外部 viseme 生成
-> 表情/情绪曲线生成
-> 驱动 MetaHuman 或自定义数字人
```

验收标准：

- 不播放音频也能拿到 wav 文件路径。
- 普通播放链路和数字人链路互不阻塞。
- 后续增加口型/表情字段时，不需要推翻现有蓝图入口。

## 8. P6：长文本分段与队列播放

目标：让插件能够处理大段文字输入，自动拆分成多个稳定的 TTS 段落，并按队列顺序生成、播放和输出事件。

当前结论：

- 大段文字不应该一次性发给 OmniVoice。
- 长文本要拆成短句或短段，每段单独生成 wav。
- 队列播放要能和字幕、数字人口型、表情系统对齐。
- 这项能力不是当前 MVP 必需，但属于插件产品化的重要能力。

后续任务：

- [x] 设计长文本分段规则：标点切分、换行切分、最大字符数、最小段落合并。
- [x] 新增 `FLocalTTSTextSegment`，记录段序号、段文本、原文范围、推荐 Duration。
- [x] 新增 `FLocalTTSLongTextRequest`，封装原始长文本、模式、语言、音色参数和分段策略。
- [x] 新增 `ULocalTTSLongTextQueue`，负责分段生成、顺序播放、暂停、继续、停止和跳段。
- [ ] 新增队列蓝图异步节点，例如 `Speak Long Text Async` 和 `Generate Long Text Speech Events Async`。
- [x] 增加队列回调：队列开始、段落开始、段落生成完成、段落播放完成、队列完成、队列失败。
- [x] 长文本队列中每段都输出 `FLocalTTSSpeechEvent`，方便字幕和数字人系统消费。
- [x] 支持中途 `Stop` 清理当前播放和未完成请求。
- [x] 设计 cache 清理策略，避免长文本生成大量 wav 长期残留。
- [x] 文档明确长文本应优先按“生成段落语音事件”和“播放段落 WAV”拆分消费。

当前实现说明：

- `ULocalTTSLongTextQueue` 已支持 `StartSpeakQueue`、`StartGenerateQueue`、`PauseQueue`、`ResumeQueue`、`SkipToNext`、`StopQueue`。
- 播放中暂停会暂停当前音频；生成中暂停会等待当前段生成完成后停在段落边界。
- 生成中跳段和停止属于软控制，当前服务端推理请求不会被立即取消，但队列会忽略已跳过或已停止的段结果。
- 当前还没有额外封装成长文本专用蓝图异步节点，蓝图需要先创建并保存 `ULocalTTSLongTextQueue` 对象引用。

验收标准：

- 输入一段包含多句的中文文本，可以自动拆成多个请求。
- 队列按顺序生成并播放，不会并发覆盖 `LastTTSResponse` 或播放组件。
- 任意一段失败时能显示段序号、段文本和错误原因。
- 字幕/数字人系统能拿到当前段文本、request_id、wav_path、段序号。

建议链路：

```text
LongText
-> FLocalTTSLongTextRequest
-> 分段得到多个 FLocalTTSTextSegment
-> ULocalTTSLongTextQueue 顺序调用 Generate/Speak
-> 每段输出 FLocalTTSSpeechEvent
-> 播放器 / 字幕 / 数字人系统按段消费
```
