# 下一阶段任务清单

## 1. 当前可用状态

当前项目已经完成 MVP 验收闭环：

- UE5.6 工程可以加载 `LocalTTS` 插件。
- UE 可以启动本地 FastAPI TTS 服务。
- 本地服务可以加载 `OmniVoice`。
- UE 可以调用 `/health`。
- UE 可以调用 `/tts`。
- 服务可以生成 wav 文件。
- UE 可以读取 `LastGeneratedWavPath`。

最近一次有效结果示例：

```text
D:\project\UETTsProject\Services\tts_service\cache\req_000003.wav
```

## 2. 下一步优先级

下一阶段优先做产品化，不急着做 MetaHuman、口型、流式播放。

推荐顺序：

1. 蓝图正式接口整理。
2. 错误处理与状态机。
3. 音频播放稳定性。
4. OmniVoice 参数体验。
5. 编译、安装、交付流程。

## 7. P5：数字人联动预留

目标：插件不只服务于“播放音频”，还要能服务于数字人口型、表情和表演驱动。

当前结论：

- 短期不直接实现 MetaHuman、口型识别或表情系统。
- 现在必须先把“语音生成”和“音频播放”解耦，避免后面数字人只能从播放器里反向取数据。
- `Generate Local TTS Async` 是数字人链路的推荐入口：它只生成 wav 和响应数据，不自动播放。
- `Speak Local TTS Async` 继续面向普通 2D 播放。
- `Speak Local TTS At Actor Async` 面向场景内 Actor 的 3D 播放。

后续任务：

- [x] 增加只生成 wav、不自动播放的蓝图节点。
- [x] 增加统一语音事件结构体 `FLocalTTSSpeechEvent`。
- [x] 在语音事件中保留 `RequestId`、`WavPath`、`Text`、`DurationSeconds` 等字段。
- [x] 预留 Viseme 口型轨道字段。
- [x] 预留 Emotion 表情轨道字段。
- [x] 异步语音节点增加 `OnSpeechEventReady` 回调。
- [ ] 设计数字人驱动接口，例如 `ULocalTTSDigitalHumanDriver`。
- [ ] 调研 UE/MetaHuman 可接入口型方案，再决定是否接第三方 lip sync 插件或自研分析。

建议链路：

```text
OmniVoice 生成 wav
-> Generate Local TTS Async 返回响应
-> 数字人系统读取 wav/text/request_id
-> 口型分析或外部 viseme 生成
-> 表情/情绪曲线生成
-> 驱动 MetaHuman 或自定义数字人
```

验收标准：

- 不播放音频也能拿到 wav 文件路径。
- 普通播放链路和数字人链路互不阻塞。
- 后续增加口型/表情字段时，不需要推翻现有蓝图入口。
