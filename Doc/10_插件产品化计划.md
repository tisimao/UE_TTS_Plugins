# 插件产品化计划

## 1. 当前结论

当前 `LocalTTS` 已经完成第一版 MVP 闭环：

- UE5.6 插件可以启动本地 TTS 服务。
- UE 可以通过 HTTP 调用本地 `OmniVoice` 服务。
- 服务可以返回 `/health` 健康状态。
- 服务可以通过 `/tts` 生成 wav 文件。
- UE 可以拿到 `LastGeneratedWavPath`。
- UE 日志已经出现 `LocalTTS audio playback started.`。
- 实际测试已确认 UE 内播放链路可用，输入文本可以按需朗读。
- 实际测试已确认 `design` 模式参数链路可用，音色可按项目需求继续匹配。

因此当前状态定义为：

> MVP 和现阶段语音接口已验收通过，但还不是产品化完成。

## 2. 产品化目标

产品化不是继续堆功能，而是把“能跑”变成“稳定、易用、可维护、可交付”。

产品化后的插件应满足：

- 蓝图用户不需要理解本地服务细节。
- 第一次使用时能清楚知道服务是否启动、模型是否可用、请求是否成功。
- 出错时能看到明确原因和下一步操作。
- 项目迁移、路径变化、UE 重启后仍能稳定工作。
- 常用 TTS 参数能通过清晰的蓝图结构体配置。
- 服务日志、UE 日志、运行时结果字段能互相对应，便于排查。
- 后续可以扩展 `auto`、`design`、`clone`、长文本分段队列、流式播放、MetaHuman 口型等能力。

## 3. 产品化阶段规划

### P0：MVP 验收收尾

目标：确认当前闭环稳定，不再处于“偶然跑通”状态。

任务：

- [x] UE 能启动本地服务。
- [x] UE 能请求 `/health`。
- [x] UE 能请求 `/tts`。
- [x] 服务能生成 wav。
- [x] UE 能拿到 `LastGeneratedWavPath`。
- [x] 人工确认生成音频内容正确。
- [x] 人工确认 UE 内播放音量、时机、重复播放行为符合预期。
- [x] 人工确认 `design` 模式配置链路可用。

验收标准：

- 连续点击 `Speak` 3 次，每次都能生成新的 `req_xxx.wav`。
- `LastErrorMessage` 为空。
- `LastGeneratedWavPath` 指向真实存在的 wav 文件。
- 服务日志中每次请求都有 `request started` 和 `request finished`。
- UE 内可以听到播放声音，且文本内容按输入朗读。
- `design` 模式可以通过合法 `Instruct` 标签完成生成和播放。

### P1：蓝图接口整理

目标：让插件从“测试 Actor 可用”变成“项目蓝图可用”。

任务：

- [x] 明确正式蓝图入口：`StartLocalTTS`、`CheckLocalTTSHealth`、`SpeakTextAsync`、`StopSpeaking`。
- [x] 给蓝图节点补齐中文/英文友好显示名。
- [x] 将测试 Actor 保留为示例，不作为正式业务入口。
- [ ] 整理 `FLocalTTSSpeakRequest` 字段显示名、默认值、Tooltip。
- [ ] 整理 `FLocalTTSTTSResponse` 字段显示名、Tooltip。
- [x] 增加 `LastGeneratedWavPath` 这类调试字段到示例 Actor。

验收标准：

- 用户只通过蓝图节点也能完成一次 TTS 请求。
- 不需要拖测试 Actor，也能在任意蓝图里调用 TTS。
- 节点和参数命名清晰，不依赖阅读 C++ 源码。

### P2：错误处理与状态机

目标：把“失败了”变成“知道为什么失败”。

任务：

- [ ] 统一错误码：服务未启动、模型未加载、请求参数错误、HTTP 超时、wav 加载失败、播放失败。
- [ ] `ULocalTTSSubsystem` 增加明确状态：`Stopped`、`Starting`、`Ready`、`Busy`、`Error`。
- [ ] 防止重复启动服务。
- [ ] 防止重复并发 Speak 导致状态覆盖。
- [ ] 对 HTTP 502、连接失败、超时分别给出不同错误信息。
- [x] 在 UE 日志中输出 request_id，方便和服务日志对齐。

验收标准：

- 常见错误都能在 `LastErrorMessage` 或蓝图 `OnError` 中看到可读原因。
- 服务未启动时点击 `Speak` 可以自动启动或给出明确提示。
- 重复点击 `Start` 不会重复拉起多个服务进程。

### P3：音频播放产品化

目标：让 wav 播放成为稳定能力，而不是只验证路径。

任务：

- [x] 明确 wav 加载策略：运行时文件加载、缓存、释放。
- [x] 播放完成后触发稳定回调。
- [x] 支持停止当前播放。
- [x] 支持选择播放目标：2D 播放、Actor 位置播放、AudioComponent 播放。
- [x] 处理没有音频设备、音频文件不存在、wav 格式不支持等情况。
- [x] 记录播放开始、播放完成、播放失败日志。

当前实现说明：

- 当前运行时路径是“读取服务端落盘 wav -> 校验 -> 转成 `USoundWaveProcedural` -> UE 播放”。
- 当前播放链路正式支持 `16-bit PCM` 的 mono/stereo wav。
- 服务端 `cache` 目录已增加基础自动清理策略，默认保留最近 `50` 个 wav，避免持续堆积。

验收标准：

- 同一个文本生成后可以正常播放。
- 多次请求不会残留无效 AudioComponent。
- `Stop` 可以中断当前播放。
- 蓝图能收到 `OnAudioReady`、`OnStarted`、`OnFinished`、`OnError`。
- 缺失 wav、坏 wav、无音频设备场景都能返回明确错误。

正式 UI 接入建议：

- `OnStateChanged` 用于统一驱动 UI 状态机，减少蓝图里手写布尔量和分支。
- `OnStarted` 用于显示“思考中 / 生成中”。
- `OnAudioReady` 用于切到“可播放 / 播放中”。
- `OnFinished` 用于恢复按钮和结束播放提示。
- `OnError` 用于展示错误并恢复交互。
- 不建议仅依赖 `OnSucceeded` 切到“播放中”，因为它更接近“服务端已生成成功”，而不是“音频已准备完成”。

### P4：OmniVoice 参数体验

目标：让 `OmniVoice` 的能力以 UE 用户能理解的方式暴露。当前 `auto` 和 `design` 接口链路已通过实际测试，具体音色效果需要按项目角色、场景和声音风格继续匹配。

任务：

- [ ] 梳理 `auto` 模式参数。
- [ ] 梳理 `design` 模式参数。
- [ ] 梳理 `clone` 模式参数。
- [ ] 对 `ReferenceAudioPath`、`ReferenceText`、`Instruct` 增加校验与提示。
- [ ] 增加参数预设：中文默认、英文默认、快速测试、克隆测试。
- [ ] 明确 `language_id` 可选值。
- [ ] 明确 `speed`、`duration` 的合法范围。

验收标准：

- 用户能在 Details 或蓝图里理解每个参数的用途。
- 参数错误时不会静默失败。
- `auto`、`design`、`clone` 至少各有一条可复现实例。

### P5：交付与工程化

目标：让项目能被迁移、编译、交付。

任务：

- [ ] 固定 UE5.6 工程编译流程。
- [ ] 保留 `Build_UE56_TTSHost.bat` 作为一键编译入口。
- [ ] 解决或绕开 UE 引擎侧 `Magick.NET` NuGet 警告阻断。
- [ ] 明确 Python 环境初始化方式。
- [ ] 明确模型缓存位置与首次下载说明。
- [ ] 增加服务启动失败时的安装检查。
- [x] 增加 README 快速启动步骤。
- [ ] 明确哪些目录应该提交 git，哪些目录不提交。

验收标准：

- 新机器按照文档可以启动服务并跑通一次 TTS。
- 工程重新生成后不会因为无关插件阻塞。
- 路径变化后可以通过配置修正。

### P6：长文本分段与队列播放

目标：让插件能稳定处理大段文本，而不是要求用户一次只输入短句。

当前结论：

- 不建议把一大段文字一次性发给 OmniVoice。
- 长文本应该先按句子或段落切分，再逐段生成 wav。
- 播放层需要按顺序播放每段音频，并能暂停、继续、停止。
- 数字人链路也需要按分段输出字幕、语音事件、口型和表情时间线。

任务：

- [x] 新增长文本分段策略，例如按中文句号、问号、感叹号、换行和最大字符数切分。
- [x] 新增长文本请求结构，例如 `FLocalTTSLongTextRequest`。
- [x] 新增分段结果结构，例如 `FLocalTTSTextSegment` 和 `FLocalTTSSegmentSpeechEvent`。
- [x] 新增队列播放类，例如 `ULocalTTSLongTextQueue`，负责顺序生成、顺序播放和状态管理。
- [x] 支持队列状态：Idle、Segmenting、Generating、Playing、Paused、Stopped、Finished、Error。
- [x] 支持队列操作：Start、Pause、Resume、Stop、SkipToNext。
- [x] 支持每段回调：OnSegmentStarted、OnSegmentGenerated、OnSegmentFinished。
- [x] 支持整段回调：OnQueueStarted、OnQueueFinished、OnQueueStopped、OnQueueError。
- [x] 为字幕系统输出当前段文本、段序号、request_id、wav_path。
- [x] 为数字人系统输出每段 `FLocalTTSSpeechEvent`，后续可串进口型和表情轨道。
- [x] 设计缓存清理策略，避免大段文本生成大量 wav 后无限堆积。

当前实现说明：

- 长文本队列内部统一先逐段 `Generate`，再由队列决定是否播放，这样暂停、继续、跳段和字幕/数字人事件更容易对齐。
- 播放中暂停会真正暂停当前 `AudioComponent`，继续会恢复当前音频。
- 生成中暂停会等待当前段生成完成后停在段落边界；当前还不能立即取消服务端正在进行的推理。
- 生成中跳段属于软跳过：当前请求返回后丢弃该段结果并进入下一段。

验收标准：

- 输入一段多句中文文本，可以自动拆分成多个 TTS 请求。
- 每段 wav 按顺序生成并播放，不会互相覆盖状态。
- `Stop` 可以停止整个队列，而不是只停止当前 wav。
- 某一段失败时能明确指出失败段落和错误原因。
- 字幕或数字人系统能知道当前正在播放第几段、播放文本是什么、对应 wav 在哪里。

## 4. 推荐下一步

当前 P1、P2、P3 和 P6 的核心骨架已经落地。下一步建议围绕“长文本能力可测试、可演示、可交付”继续收口，不建议立刻进入 MetaHuman、真实口型或流式播放。

推荐顺序：

1. 在 UE 内按 `13_明日测试步骤.md` 和 `16_轻量DemoUI设计.md` 做人工测试。
2. 搭建轻量 Demo UI 和 Demo 关卡，`BP_LocalTTS_DemoController` 建议继承 `ALocalTTSDemoController`，验证长文本队列按钮、状态、段落事件是否符合预期。
3. 根据人工测试结果修正长文本分段细节和暂停、继续、跳段体验。
4. 再考虑封装长文本专用异步蓝图节点，减少 UI 手动保存队列对象的步骤。
5. 最后再进入数字人口型、表情驱动和 MetaHuman 接入。

原因：

- 当前长文本队列已经能分段、逐段生成、顺序播放、暂停、继续、跳段和停止。
- `ALocalTTSDemoController` 已经把 Demo UI 需要的按钮操作、状态文本、最近结果和事件日志集中到示例层，后续人工搭 UI 时不需要从零绑定所有底层回调。
- 现在最需要确认的是实际 UE 操作体验，而不是继续扩新功能。
- Demo UI 会反向检验蓝图节点是否好用，能提前发现正式产品接口是否需要再封装。
- 数字人联动依赖稳定的段落事件、request_id、wav_path 和当前段文本，必须先把长文本队列验稳。

## 5. 当前已知风险

- `OmniVoice` 的 ASR warmup 可能因为网络或缓存失败，但当前服务可以 fallback 到无 ASR 模式。
- UE Details 面板容易选中编辑器原始 Actor，而不是 PIE 运行时 Actor，导致看不到运行时字段变化。
- VS 生成整个解决方案会触发 UE 自带 C# 工具项目的 NuGet 漏洞警告，建议使用 `Build_UE56_TTSHost.bat`。
- 当前测试 Actor 会在 `bSpeakOnBeginPlay=true` 时自动发声，手动测试时可能产生重复请求。
- 当前服务端已增加按数量自动清理旧 wav 的基础策略；后续仍可继续扩展为按总大小、按时间或按长文本队列批次清理。
- 长文本队列的生成中暂停、跳段和停止目前属于软控制；服务端正在推理的请求不会被立即取消，但队列会忽略已跳过或已停止的段结果。
- 当前还没有长文本专用异步蓝图节点；如果直接用底层队列，Demo UI 需要保存 `ULocalTTSLongTextQueue` 对象引用。使用 `ALocalTTSDemoController` 时，该引用由控制器保存。

## 6. 产品化完成定义

当以下条件全部满足时，可以认为 `LocalTTS` 插件产品化完成：

- 蓝图节点可独立使用，不依赖测试 Actor。
- 服务启动、健康检查、语音生成、播放、停止都有稳定回调。
- 所有常见失败都有明确错误信息。
- `auto`、`design`、`clone` 模式有可复现实例。
- 文档包含安装、编译、运行、调试、常见问题。
- 大段文本可以通过分段队列稳定生成、播放、停止，并向字幕/数字人系统输出分段事件。
- 新机器可以按文档完成一次端到端 TTS。
- 插件代码、服务代码、文档均遵守“类名即文件名”的约定。
