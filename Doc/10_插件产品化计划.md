# 插件产品化计划

## 1. 当前结论

当前 `LocalTTS` 已经完成第一版 MVP 闭环：

- UE5.6 插件可以启动本地 TTS 服务。
- UE 可以通过 HTTP 调用本地 `OmniVoice` 服务。
- 服务可以返回 `/health` 健康状态。
- 服务可以通过 `/tts` 生成 wav 文件。
- UE 可以拿到 `LastGeneratedWavPath`。
- UE 日志已经出现 `LocalTTS audio playback started.`。

因此当前状态定义为：

> MVP 已跑通，但还不是产品化完成。

## 2. 产品化目标

产品化不是继续堆功能，而是把“能跑”变成“稳定、易用、可维护、可交付”。

产品化后的插件应满足：

- 蓝图用户不需要理解本地服务细节。
- 第一次使用时能清楚知道服务是否启动、模型是否可用、请求是否成功。
- 出错时能看到明确原因和下一步操作。
- 项目迁移、路径变化、UE 重启后仍能稳定工作。
- 常用 TTS 参数能通过清晰的蓝图结构体配置。
- 服务日志、UE 日志、运行时结果字段能互相对应，便于排查。
- 后续可以扩展 `auto`、`design`、`clone`、流式播放、MetaHuman 口型等能力。

## 3. 产品化阶段规划

### P0：MVP 验收收尾

目标：确认当前闭环稳定，不再处于“偶然跑通”状态。

任务：

- [x] UE 能启动本地服务。
- [x] UE 能请求 `/health`。
- [x] UE 能请求 `/tts`。
- [x] 服务能生成 wav。
- [x] UE 能拿到 `LastGeneratedWavPath`。
- [ ] 人工确认生成音频内容正确。
- [ ] 人工确认 UE 内播放音量、时机、重复播放行为符合预期。

验收标准：

- 连续点击 `Speak` 3 次，每次都能生成新的 `req_xxx.wav`。
- `LastErrorMessage` 为空。
- `LastGeneratedWavPath` 指向真实存在的 wav 文件。
- 服务日志中每次请求都有 `request started` 和 `request finished`。

### P1：蓝图接口整理

目标：让插件从“测试 Actor 可用”变成“项目蓝图可用”。

任务：

- [ ] 明确正式蓝图入口：`StartLocalTTS`、`CheckLocalTTSHealth`、`SpeakTextAsync`、`StopSpeaking`。
- [ ] 给蓝图节点补齐中文/英文友好显示名。
- [ ] 将测试 Actor 保留为示例，不作为正式业务入口。
- [ ] 整理 `FLocalTTSSpeakRequest` 字段显示名、默认值、Tooltip。
- [ ] 整理 `FLocalTTSTTSResponse` 字段显示名、Tooltip。
- [ ] 增加 `LastGeneratedWavPath` 这类调试字段到示例 Actor。

验收标准：

- 用户只通过蓝图节点也能完成一次 TTS 请求。
- 不需要拖测试 Actor，也能在任意蓝图里调用 TTS。
- 节点和参数命名清晰，不依赖阅读 C++ 源码。

### P2：错误处理与状态机

目标：把“失败了”变成“知道为什么失败”。

任务：

- [ ] 统一错误码：服务未启动、模型未加载、请求参数错误、HTTP 超时、wav 加载失败、播放失败。
- [ ] `ULocalTTSSubsystem` 增加明确状态：`Stopped`、`Starting`、`Ready`、`Busy`、`Error`。
- [ ] 防止重复启动服务。
- [ ] 防止重复并发 Speak 导致状态覆盖。
- [ ] 对 HTTP 502、连接失败、超时分别给出不同错误信息。
- [ ] 在 UE 日志中输出 request_id，方便和服务日志对齐。

验收标准：

- 常见错误都能在 `LastErrorMessage` 或蓝图 `OnError` 中看到可读原因。
- 服务未启动时点击 `Speak` 可以自动启动或给出明确提示。
- 重复点击 `Start` 不会重复拉起多个服务进程。

### P3：音频播放产品化

目标：让 wav 播放成为稳定能力，而不是只验证路径。

任务：

- [ ] 明确 wav 加载策略：运行时文件加载、缓存、释放。
- [ ] 播放完成后触发稳定回调。
- [ ] 支持停止当前播放。
- [ ] 支持选择播放目标：2D 播放、Actor 位置播放、AudioComponent 播放。
- [ ] 处理没有音频设备、音频文件不存在、wav 格式不支持等情况。
- [ ] 记录播放开始、播放完成、播放失败日志。

验收标准：

- 同一个文本生成后可以正常播放。
- 多次请求不会残留无效 AudioComponent。
- `Stop` 可以中断当前播放。
- 蓝图能收到 `OnAudioReady`、`OnStarted`、`OnFinished`、`OnError`。

### P4：OmniVoice 参数体验

目标：让 `OmniVoice` 的能力以 UE 用户能理解的方式暴露。

任务：

- [ ] 梳理 `auto` 模式参数。
- [ ] 梳理 `design` 模式参数。
- [ ] 梳理 `clone` 模式参数。
- [ ] 对 `ReferenceAudioPath`、`ReferenceText`、`Instruct` 增加校验与提示。
- [ ] 增加参数预设：中文默认、英文默认、快速测试、克隆测试。
- [ ] 明确 `language_id` 可选值。
- [ ] 明确 `speed`、`duration` 的合法范围。

验收标准：

- 用户能在 Details 或蓝图里理解每个参数的用途。
- 参数错误时不会静默失败。
- `auto`、`design`、`clone` 至少各有一条可复现实例。

### P5：交付与工程化

目标：让项目能被迁移、编译、交付。

任务：

- [ ] 固定 UE5.6 工程编译流程。
- [ ] 保留 `Build_UE56_TTSHost.bat` 作为一键编译入口。
- [ ] 解决或绕开 UE 引擎侧 `Magick.NET` NuGet 警告阻断。
- [ ] 明确 Python 环境初始化方式。
- [ ] 明确模型缓存位置与首次下载说明。
- [ ] 增加服务启动失败时的安装检查。
- [ ] 增加 README 快速启动步骤。
- [ ] 明确哪些目录应该提交 git，哪些目录不提交。

验收标准：

- 新机器按照文档可以启动服务并跑通一次 TTS。
- 工程重新生成后不会因为无关插件阻塞。
- 路径变化后可以通过配置修正。

## 4. 推荐下一步

下一步建议先做 P1 和 P2，不建议立刻做 MetaHuman、口型、流式播放。

推荐顺序：

1. 整理正式蓝图接口。
2. 给蓝图节点和结构体补 DisplayName / Tooltip。
3. 完善 `ULocalTTSSubsystem` 状态机。
4. 防止重复启动和重复请求。
5. 给所有错误路径补清晰错误信息。
6. 再进入音频播放产品化。

原因：

- 当前链路已经能生成 wav。
- 现在最容易卡用户的是“我该点哪里、哪里出错、结果在哪看”。
- 先把接口和错误处理做好，后续扩展语音参数、口型、MetaHuman 会更稳。

## 5. 当前已知风险

- `OmniVoice` 的 ASR warmup 可能因为网络或缓存失败，但当前服务可以 fallback 到无 ASR 模式。
- UE Details 面板容易选中编辑器原始 Actor，而不是 PIE 运行时 Actor，导致看不到运行时字段变化。
- VS 生成整个解决方案会触发 UE 自带 C# 工具项目的 NuGet 漏洞警告，建议使用 `Build_UE56_TTSHost.bat`。
- 当前测试 Actor 会在 `bSpeakOnBeginPlay=true` 时自动发声，手动测试时可能产生重复请求。
- 当前服务端 cache 文件会持续增长，需要后续设计清理策略。

## 6. 产品化完成定义

当以下条件全部满足时，可以认为 `LocalTTS` 插件产品化完成：

- 蓝图节点可独立使用，不依赖测试 Actor。
- 服务启动、健康检查、语音生成、播放、停止都有稳定回调。
- 所有常见失败都有明确错误信息。
- `auto`、`design`、`clone` 模式有可复现实例。
- 文档包含安装、编译、运行、调试、常见问题。
- 新机器可以按文档完成一次端到端 TTS。
- 插件代码、服务代码、文档均遵守“类名即文件名”的约定。
