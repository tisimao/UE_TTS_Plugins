# 数字人联动设计

## 1. 设计目标

`LocalTTS` 插件不能只被设计成“播放 wav 的工具”。它后续还需要服务数字人系统，用语音结果驱动口型、表情和表演节奏。

因此当前架构要拆成三条链路：

- 语音生成：负责调用 OmniVoice，得到 wav 和响应数据。
- 音频播放：负责在 UE 中播放 2D 或 3D 声音。
- 数字人驱动：负责把语音结果转换成口型、表情和动画控制数据。

当前优先落地的是第一条和第二条。第三条先预留接口和文档，不急着直接绑定 MetaHuman。

## 2. 当前已落地能力

- `Generate Local TTS Async`：只生成 wav，不自动播放，适合数字人系统接入。
- `Speak Local TTS Async`：生成 wav 后自动 2D 播放。
- `Speak Local TTS At Actor Async`：生成 wav 后绑定到指定 Actor 位置播放。
- `Get Last Local TTS WAV Path`：读取最近一次生成的 wav 路径。
- `FLocalTTSSpeechEvent`：统一语音事件结构，已经预留口型帧和表情帧。
- `OnSpeechEventReady`：语音异步节点生成成功后广播，数字人系统优先监听这个回调。

## 3. 推荐运行链路

普通播放链路：

```text
蓝图调用 Speak Local TTS Async
-> ULocalTTSSpeakAsyncAction
-> ULocalTTSSubsystem
-> FastAPI /tts
-> OmniVoice 生成 wav
-> ULocalTTSAudioPlayer 播放 wav
```

数字人链路：

```text
蓝图调用 Generate Local TTS Async
-> ULocalTTSSpeakAsyncAction
-> ULocalTTSSubsystem
-> FastAPI /tts
-> OmniVoice 生成 wav
-> 返回 wav path / text / request_id
-> 数字人系统读取结果
-> 生成或读取口型轨道
-> 生成或读取表情轨道
-> 驱动 MetaHuman 或自定义数字人
```

## 4. 后续建议新增类

以下类都遵守“类名就是文件名”的约定。

| 类名 | 文件 | 作用 |
| --- | --- | --- |
| `FLocalTTSSpeechEvent` | `FLocalTTSSpeechEvent.h` | 统一保存一次语音生成事件，包括文本、wav 路径、请求 ID、时长，以及未来的口型和表情轨道。 |
| `FLocalTTSVisemeFrame` | `FLocalTTSVisemeFrame.h` | 表示某一时间点的口型帧，例如时间、口型名称、权重。 |
| `FLocalTTSEmotionFrame` | `FLocalTTSEmotionFrame.h` | 表示某一时间点的表情帧，例如时间、表情类型、强度。 |
| `ULocalTTSDigitalHumanDriver` | `ULocalTTSDigitalHumanDriver.h/.cpp` | 数字人驱动基类，负责接收语音事件并驱动目标 Actor。 |
| `ULocalTTSLipSyncAnalyzer` | `ULocalTTSLipSyncAnalyzer.h/.cpp` | 口型分析器接口，后续可以接第三方 lip sync、MetaHuman Animator 或自研分析。 |
| `ULocalTTSEmotionDriver` | `ULocalTTSEmotionDriver.h/.cpp` | 表情驱动接口，后续根据文本、音频能量或模型结果生成表情曲线。 |
| `ULocalTTSMetahumanDriver` | `ULocalTTSMetahumanDriver.h/.cpp` | 面向 MetaHuman 的具体实现，负责把口型和表情轨道写入 MetaHuman 可用的控制入口。 |

## 5. 当前不直接做的事情

- 不在 P3 阶段直接绑定 MetaHuman。
- 不在播放器里塞口型逻辑。
- 不把数字人逻辑写死到 `ULocalTTSAudioPlayer`。
- 不假设 OmniVoice 当前一定能直接返回 viseme 或 emotion 数据。

这样做的好处是：就算后面口型方案换成第三方插件、MetaHuman 官方方案，或者服务端新增 viseme 返回，UE 插件现有蓝图入口也不用推翻。

## 6. 下一步落地建议

第一步已经完成 `FLocalTTSSpeechEvent`，把一次语音生成结果从 `FLocalTTSTTSResponse` 扩展成更适合业务使用的事件结构。

第二步建议新增 `Generate Local TTS Speech Event Async`，让蓝图直接以 `FLocalTTSSpeechEvent` 作为主输出，而不是只监听现有节点的事件回调。

第三步再实现 `ULocalTTSLipSyncAnalyzer` 的空接口和模拟数据，先跑通“有口型帧可以驱动”的链路。

第四步根据实际数字人方案决定是否接 MetaHuman、Live Link、Control Rig 或第三方 lip sync 插件。
