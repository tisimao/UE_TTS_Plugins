# 语音生成与播放拆分设计

## 1. 目标

当前 `LocalTTS` 不再把“请求模型生成 WAV”和“在 UE 中播放 WAV”强绑定为一个动作。推荐把链路拆成两层：

```text
生成层：文本 -> WAV / TTS Response / Speech Event
播放层：WAV / TTS Response / Speech Event -> UE 播放
```

这样 UI 可以先生成声音，再决定什么时候播放、播放哪一条、是否重播旧声音，以及是否把 WAV 交给字幕、数字人或后处理系统。

## 2. 推荐蓝图入口

生成入口：

| 节点 | 作用 |
| --- | --- |
| `仅生成 Local TTS` | 请求本地服务生成 WAV，不自动播放。推荐作为正式 UI 的主生成入口。 |
| `生成并播放 Local TTS` | 兼容旧流程。内部仍是生成成功后自动接播放。适合快速测试，不建议作为复杂 UI 的唯一入口。 |
| `开始长文本仅生成` | 长文本分段后只生成每段语音事件，不自动播放。 |
| `开始长文本生成并播放` | 长文本分段后生成并顺序播放，适合演示。 |

播放入口：

| 节点 | 作用 |
| --- | --- |
| `播放 Local TTS WAV` | 输入一个 WAV 路径并播放，不请求 TTS 服务。 |
| `播放最新 Local TTS WAV` | 播放最近一次生成成功的 WAV，不请求 TTS 服务。 |
| `播放 Local TTS 响应` | 播放一个 `FLocalTTSTTSResponse` 中的 WAV，适合从历史列表重播旧声音。 |
| `播放 Local TTS 语音事件` | 播放一个 `FLocalTTSSpeechEvent` 中的 WAV，适合长文本、字幕和数字人链路。 |

调试和历史入口：

| 节点 | 作用 |
| --- | --- |
| `获取最近 Local TTS 语音结果` | 获取最近一次生成响应。 |
| `获取最近 Local TTS WAV 路径` | 获取最近一次生成的 WAV 路径。 |
| `获取 Local TTS 语音历史` | 获取最近生成成功的响应列表，当前最多保留 20 条。 |

## 3. 单句 UI 推荐流程

推荐把单句区按钮拆成：

```text
应用音色配置
生成 WAV
播放最新
生成并播放
停止播放
```

其中：

- `应用音色配置` 调 `Demo 应用音色配置`，把固定 `Mode/Instruct/Speed/Duration` 写入单句和长文本模板。
- `生成 WAV` 调 `仅生成 Local TTS`。
- `播放最新` 调 `播放最新 Local TTS WAV`。
- `生成并播放` 可以继续调 `生成并播放 Local TTS`，也可以在 `仅生成 Local TTS.OnAudioReady` 后手动接 `播放 Local TTS 响应`。
- `停止播放` 调 `停止 Local TTS 播放`。

推荐状态拆分：

```text
生成状态：Started / WaitingForService / Generating / AudioReady / Finished / Error
播放状态：Started / AudioReady / Playing / Finished / Error
```

UI 不应该只用一个 `Busy` 禁用所有按钮。更合适的是：

- 生成中：禁用“生成 WAV”和“生成并播放”。
- 播放中：禁用“播放最新”，启用“停止播放”。
- 播放中仍可允许下一条“生成 WAV”，但当前服务端仍只允许一个 TTS 生成请求同时进行。

测试阶段建议默认使用：

```text
Mode = design
LanguageId = zh
Instruct = female, young adult, chinese accent, moderate pitch
Speed = 1.0
Duration = 0.0
```

不要默认用 `auto` 做稳定验收，因为它更容易出现音色随机变化。

## 4. 播放旧声音

`ULocalTTSSubsystem` 会保存最近 20 条生成成功的 `FLocalTTSTTSResponse`。

蓝图流程：

```text
获取 Local TTS 语音历史
-> 选择某一条 Response
-> 播放 Local TTS 响应
```

当前历史只保存在运行期内，重启游戏后会丢失。如果需要长期保存旧声音，后续应增加“保存到语音库”功能，把 WAV 从服务 cache 复制到：

```text
Saved/LocalTTS/Library
```

不要长期依赖：

```text
Saved/LocalTTS/cache
```

因为服务端 cache 会按上限清理旧 WAV。

## 5. 长文本推荐流程

长文本更适合按“先准备下一段，再播放/切换”的思路组织：

```text
当前段播放中
-> 下一段提前生成 WAV
-> 当前段结束后播放下一段
```

如果用户点击“下一段”，推荐语义是：

```text
不打断当前播放
-> 优先预生成下一段
-> 当前段自然结束后切到已准备好的下一段
```

当 UI 需要直接重播某一段时：

```text
段落已生成事件 SegmentSpeechEvent
-> 播放 Local TTS 语音事件
```

## 6. 数字人和字幕链路

数字人、字幕和表情系统应优先消费 `FLocalTTSSpeechEvent`，而不是直接依赖播放器。

推荐链路：

```text
仅生成 Local TTS
-> OnSpeechEventReady
-> 字幕系统读取 Text
-> 数字人系统读取 WavPath / Text / RequestId
-> 口型或表情分析
-> 需要播放时再调用播放节点
```

这样即使不播放音频，也能先得到 WAV 和文本元数据。

## 7. 当前实现说明

本轮已落地：

- `ULocalTTSPlayWavAsyncAction`
- `播放 Local TTS WAV`
- `播放最新 Local TTS WAV`
- `播放 Local TTS 响应`
- `播放 Local TTS 语音事件`
- `ULocalTTSSubsystem::PlayWavPath`
- `ULocalTTSSubsystem::PlayLastSpeech`
- `ULocalTTSSubsystem` 最近 20 条 `FLocalTTSTTSResponse` 历史
- `获取 Local TTS 语音历史`

保留兼容：

- `生成并播放 Local TTS`
- `生成并在 Actor 位置播放 Local TTS`
- `仅生成 Local TTS`

后续建议：

- Demo UI 改成“生成”和“播放”两个明确区域。
- 增加历史列表 Row，用 `播放 Local TTS 响应` 重播旧声音。
- 增加“保存到语音库”功能，把重要 WAV 从 cache 复制到长期目录。
