# 项目目录结构

## 1. 仓库根目录

```text
UETTsProject
├─ Doc
├─ UE56_TTSHost
└─ Services
```

## 2. 目录说明

### `Doc`

用于保存需求、方案、接口、计划文档。

### `UE56_TTSHost`

用于开发和测试插件的 Unreal Engine 5.6 C++ 宿主工程。

说明：

- 这个工程主要用于开发插件。
- 最终真正可复用的核心内容是插件目录。

### `Services`

用于保存本地 TTS 服务及相关脚本。

## 3. UE 工程内部结构

```text
UE56_TTSHost
├─ Config
├─ Content
├─ Source
└─ Plugins
   └─ LocalTTS
      ├─ Content
      ├─ Resources
      ├─ Source
      │  ├─ LocalTTS
      │  └─ LocalTTSEditor
      └─ ThirdParty
         └─ TTSRuntime
```

## 4. 关键约定

- 插件名固定为 `LocalTTS`
- 宿主工程名固定为 `UE56_TTSHost`
- 本地服务目录固定放在 `Services/tts_service`
- 插件只负责 UE 集成，不直接保存大型模型权重到仓库

## 5. 命名约定

为方便团队成员按文档、类图和日志快速定位代码，当前仓库执行以下命名规则：

- `UCLASS`、`USTRUCT`、`UENUM` 的类型名就是文件名
- 需要实现文件的类型，`.h` 和 `.cpp` 与类型名保持一致
- 示例 Actor 也遵守同样规则，避免宿主工程和插件工程出现两套命名风格

示例：

- `ULocalTTSSpeakAsyncAction` -> `ULocalTTSSpeakAsyncAction.h/.cpp`
- `ULocalTTSLongTextQueue` -> `ULocalTTSLongTextQueue.h/.cpp`
- `FLocalTTSSegmentSpeechEvent` -> `FLocalTTSSegmentSpeechEvent.h`
- `ELocalTTSLongTextQueueState` -> `ELocalTTSLongTextQueueState.h`

## 6. 为什么需要 C++ 工程

必须建立 C++ 工程，原因如下：

- 插件需要运行时模块
- 插件需要编辑器模块
- 插件需要暴露蓝图节点
- 插件需要管理本地进程
- 插件需要处理运行时音频

因此，本项目不能以纯蓝图工程作为主开发工程。
