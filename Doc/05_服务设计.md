# 服务设计

## 1. 服务目标

`Services/tts_service` 负责承载本地 `OmniVoice` 模型运行环境。

它不是 UE 插件的一部分，但由 UE 插件自动启动和调用。

## 2. 服务职责

- 加载 `OmniVoice` 模型
- 暴露本地 HTTP 接口
- 接收 `auto`、`clone`、`design` 模式请求
- 生成 wav 音频
- 返回结果给 UE 插件

## 3. 服务目录约定

```text
Services/tts_service
├─ app
├─ cache
├─ logs
├─ models
├─ scripts
├─ requirements.txt
└─ run_server.py
```

## 4. 目录说明

### `app`

服务源码目录。

建议包含：

- `AppConfig.py`
- `FastAPIAppFactory.py`
- `HealthResponse.py`
- `OmniVoiceEngine.py`
- `RequestIdGenerator.py`
- `RequestValidator.py`
- `ServiceError.py`
- `ServiceLogger.py`
- `TTSController.py`
- `TTSRequest.py`
- `TTSResponse.py`
- `TTSService.py`
- `WavCacheManager.py`

### `cache`

保存生成后的临时 wav 文件。

规则：

- 由服务自动创建
- 可按请求编号命名
- 可后续增加清理策略

### `logs`

保存服务日志。

### `models`

保存本地模型文件和配置。

注意：

- 大模型权重默认不提交到 git

### `scripts`

保存环境初始化脚本和启动脚本。

## 5. 服务启动方式

第一版默认由 UE 插件自动启动。

建议启动流程：

1. 插件检查端口是否已有服务
2. 若没有，则启动 `run_server.py`
3. 等待服务健康检查通过
4. 再发送 TTS 请求

## 6. 服务运行方式

第一版固定为：

- 本机运行
- Windows 平台
- 本地回环地址 `127.0.0.1`
- 本地 GPU 推理

## 7. 服务接口

第一版最少需要：

- `GET /health`
- `POST /tts`

可选扩展：

- `GET /languages`
- `POST /stop`

补充说明：

- `OmniVoice` 本身不是现成的业务 HTTP 服务。
- 它提供的是 Python API、单次推理 CLI 和 Gradio Web Demo。
- UE 插件如果直接依赖这些入口，会缺少稳定的 `/health`、`/tts`、错误码约定、日志与缓存管理能力。
- 因此本项目保留一层本地服务封装，作为 UE 与模型之间的稳定边界。

## 8. 模型策略

第一版固定主路线：

- `OmniVoice`

约束：

- 支持 `auto`、`clone`、`design`
- 支持 `language_id`
- 支持 `speed`
- 支持 `duration`
- `clone` 模式下 `ref_audio` 为必填
- `clone` 模式下 `ref_text` 可选，缺省时优先交由模型侧自动转写
- 若运行环境不可用 ASR 或离线且未提前缓存 ASR 模型，则 `clone` 请求必须显式提供 `ref_text`
- 输出统一为 wav

## 9. 缓存策略

第一版采用最简单策略：

- 每次请求生成一个新 wav
- 文件保存在 `cache`
- 返回文件绝对路径
- 默认自动保留最近 `50` 个 wav，超过上限时删除更旧文件

后续再优化：

- 按总大小而不是只按数量清理旧缓存
- 保存可复用的参考提示缓存

## 10. 日志策略

日志至少要记录：

- 服务启动
- 模型加载成功或失败
- 收到的请求编号
- 请求模式
- 生成耗时
- 生成结果路径
- 异常信息
