# UE TTS 插件技术方案

## 1. 总体方案

本项目采用“UE 插件 + 本地 TTS 服务进程”的架构。

对使用者来说：

- 它是一个 UE 插件。
- 它提供蓝图节点。
- 蓝图调用后可以直接发声。

对内部实现来说：

- UE 插件负责启动和管理本地 TTS 服务进程。
- 插件通过本地 HTTP 与 TTS 服务通信。
- TTS 服务负责加载模型并执行推理。
- 插件将返回的音频加载到 UE 中并播放。

## 2. 为什么不用“模型直接嵌进 UE 插件”

- PyTorch 和 CUDA 依赖重。
- 与 UE 运行时直接耦合，调试困难。
- 模型崩溃可能拖垮 UE 进程。
- 后期升级模型成本高。
- 打包和部署复杂。

因此第一版采用“插件托管外部本地服务”的模式。

## 3. 模块划分

### 3.1 UE 插件运行时模块

负责：

- 启动本地 TTS 服务进程
- 检查服务状态
- 发送文本请求
- 接收音频结果
- 在 UE 中播放音频
- 暴露蓝图接口

### 3.2 UE 插件编辑器模块

负责：

- 提供插件设置面板
- 配置模型目录
- 配置服务端口
- 配置默认音色
- 提供调试入口

### 3.3 本地 TTS 服务

负责：

- 加载 CosyVoice 模型
- 接收文本请求
- 生成 wav 音频
- 将结果返回给 UE 插件

## 4. 通信协议

第一版采用本地 HTTP。

默认地址：

- `http://127.0.0.1:50021`

第一版接口：

- `POST /tts`

请求示例：

```json
{
  "text": "你好，这是测试语音。",
  "voice_id": "default_cn_female",
  "speed": 1.0
}
```

响应示例：

```json
{
  "ok": true,
  "wav_path": "D:/project/UETTsProject/Services/tts_service/cache/req_0001.wav",
  "duration_ms": 2100
}
```

## 5. 插件对外蓝图接口

第一版计划暴露以下节点：

- `StartLocalTTS`
- `IsLocalTTSReady`
- `SpeakTextAsync`
- `StopSpeaking`

核心节点：

- `SpeakTextAsync(Text, VoiceId, Speed)`

第一版不暴露复杂参数。

## 6. 音频策略

- 第一版只支持 wav。
- 推荐输出格式为 16-bit PCM、mono、48kHz。
- UE 侧优先做“完整音频生成后播放”。
- 第一版不做流式音频队列播放。

## 7. 第一版实现优先级

- 先打通文本到音频播放
- 再优化服务启动体验
- 再补插件设置和调试能力

## 8. 后续扩展方向

这些内容明确放到第一版之后：

- 流式 TTS
- 数字人口型驱动
- MetaHuman 集成
- 多音色切换
- 情绪参数
- 长文本自动分句
