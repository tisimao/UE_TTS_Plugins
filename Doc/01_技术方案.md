# UE TTS 插件技术方案

## 1. 总体方案

本项目采用“UE 插件 + 本地 TTS 服务进程”的架构。

对使用者来说：

- 它是一个 UE 插件。
- 它提供蓝图节点。
- 蓝图调用后可以直接发声。

对内部实现来说：

- UE 插件负责启动和管理本地 TTS 服务进程。
- 插件通过本地 HTTP 与 TTS 服务通信。
- TTS 服务负责加载 `OmniVoice` 并执行推理。
- 插件将返回的音频加载到 UE 中并播放。

## 2. 为什么不用“模型直接嵌进 UE 插件”

- PyTorch 和 CUDA 依赖重。
- 与 UE 运行时直接耦合，调试困难。
- 模型崩溃可能拖垮 UE 进程。
- 后期升级模型成本高。
- 打包和部署复杂。

因此第一版采用“插件托管外部本地服务”的模式。

## 3. 模块划分

### 3.1 UE 插件运行时模块

负责：

- 启动本地 TTS 服务进程
- 检查服务状态
- 发送语音请求
- 接收音频结果
- 在 UE 中播放音频
- 暴露蓝图接口

### 3.2 UE 插件编辑器模块

负责：

- 提供插件设置面板
- 配置模型目录
- 配置服务端口
- 配置默认模式和默认语言
- 提供调试入口

### 3.3 本地 TTS 服务

负责：

- 加载 `OmniVoice` 模型
- 接收 `auto`、`clone`、`design` 模式请求
- 调用模型生成 wav 音频
- 将结果返回给 UE 插件

## 4. 通信协议

第一版采用本地 HTTP。

默认地址：

- `http://127.0.0.1:50021`

第一版接口：

- `GET /health`
- `POST /tts`

说明：

- `OmniVoice` 官方当前提供的是 Python 推理接口、命令行工具和 Gradio Demo。
- 它没有提供一个可直接给 UE 插件长期依赖的稳定本地 HTTP 服务协议。
- 因此我们仍然需要自己封装本地服务，统一健康检查、请求格式、错误码、日志、缓存路径和进程生命周期。

请求示例：

```json
{
  "text": "你好，这是测试语音。",
  "mode": "design",
  "language_id": "zh",
  "instruct": "成年女性，普通话，自然播报",
  "duration": null,
  "speed": 1.0
}
```

响应示例：

```json
{
  "ok": true,
  "request_id": "req_000001",
  "mode": "design",
  "wav_path": "D:/project/UETTsProject/Services/tts_service/cache/req_000001.wav",
  "sample_rate": 24000,
  "duration_ms": 2100
}
```

## 5. 插件对外蓝图接口

第一版计划暴露以下节点：

- `StartLocalTTS`
- `IsLocalTTSReady`
- `SpeakTextAsync`
- `StopSpeaking`

核心节点：

- `SpeakTextAsync(Request)`

其中 `Request` 为蓝图结构体，至少包含：

- `Text`
- `Mode`
- `LanguageId`
- `ReferenceAudioPath`
- `ReferenceText`
- `Instruct`
- `Duration`
- `Speed`

这样可以避免把大量可选参数直接摊平成蓝图引脚。

## 6. 音频策略

- 第一版只支持 wav。
- 默认输出格式为 `16-bit PCM`、`mono`、`24kHz`。
- UE 侧优先做“完整音频生成后播放”。
- 第一版不做流式音频队列播放。

## 7. 第一版实现优先级

- 先打通 `auto` 模式下的文本到音频播放
- 再打通 `design` 模式
- 再打通 `clone` 模式
- 再补插件设置和调试能力

## 8. 后续扩展方向

这些内容明确放到第一版之后：

- 流式 TTS
- 数字人口型驱动
- MetaHuman 集成
- 参考音频提示缓存管理
- 长文本自动分句与长音频拼接
